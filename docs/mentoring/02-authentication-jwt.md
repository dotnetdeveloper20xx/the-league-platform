# Authentication and JWT

Let me walk you through how authentication works in this system, because it is one of those things that seems straightforward on the surface but has quite a few moving parts underneath. We use JSON Web Tokens as our authentication mechanism, and the Identity module is responsible for everything related to users, passwords, tokens, and sessions. Understanding this flow end-to-end is essential because every authenticated request in the system depends on it working correctly.

When a user wants to log in, they send a POST request to the api/v1/auth/login endpoint with their email and password. The request is handled by the AuthController, which creates a LoginCommand and sends it through MediatR. The LoginCommand handler is where the real work happens. First, it uses ASP.NET Core Identity's UserManager to find the user by email. If no user exists with that email, it returns a generic "invalid credentials" error — we never reveal whether an email exists in the system, because that would be an information disclosure vulnerability.

If the user exists, the handler checks whether the account is locked out. ASP.NET Core Identity tracks failed login attempts automatically. After five consecutive failed attempts, the account is locked for fifteen minutes. The handler checks this before even attempting to validate the password. If the account is locked, it returns a 403 response with the number of seconds remaining in the lockout period. The frontend uses this to display a countdown timer to the user.

Assuming the account is not locked, the handler validates the password using UserManager.CheckPasswordAsync. If the password is wrong, it increments the failed attempt counter (which Identity does automatically) and returns an unauthorized response. If the password is correct, it resets the failed attempt counter and proceeds to generate tokens.

The access token is a standard JWT signed with a symmetric key using HMAC-SHA256. The key is configured in appsettings.json under the Jwt:Secret setting. In development, we use a hardcoded key that is at least sixty-four characters long. In production, this would come from a secure secret store like Azure Key Vault. The token contains several claims: the user's ID (as the sub claim), their email, their role, and their clubId. That clubId claim is critical — it is what the tenant middleware reads to establish the multi-tenant context for every subsequent request.

The access token has a short lifetime of fifteen minutes. This is a deliberate security decision. If an access token is somehow compromised — intercepted in transit, leaked through a log, extracted from a compromised client — the window of exposure is very small. An attacker can only use it for fifteen minutes before it expires. This is why we pair it with a refresh token.

The refresh token is a cryptographically random string — not a JWT. It is stored in the database in the RefreshTokens table within the identity schema. Each refresh token record includes the token value, the user ID it belongs to, an expiration date (seven days from creation), and a flag indicating whether it has been used. The refresh token is returned to the client alongside the access token in the login response.

When the access token expires, the client sends the refresh token to the api/v1/auth/refresh endpoint. The handler looks up the refresh token in the database, verifies it has not expired and has not already been used, and then performs refresh token rotation. This means the old refresh token is marked as used (it can never be used again), and a brand new refresh token is generated and stored. A new access token is also generated. Both are returned to the client.

Refresh token rotation is a security measure against token theft. Imagine an attacker steals a refresh token. They use it to get a new access token. Meanwhile, the legitimate user's client also tries to use the same refresh token. The system detects that the token has already been consumed and rejects the request. This signals a potential compromise — the legitimate user is forced to log in again, and the attacker's stolen token is now useless because it was already rotated.

On the frontend side, the Angular application stores both tokens in localStorage. The auth interceptor — a functional HTTP interceptor registered in the app config — attaches the access token as a Bearer token in the Authorization header of every outgoing HTTP request. When a request returns a 401 Unauthorized response, the interceptor catches it and attempts a token refresh. If the refresh succeeds, it retries the original request with the new access token. If the refresh also fails (token expired, already used, or revoked), it clears the stored tokens and redirects the user to the login page.

This gives users a seamless experience. They log in once, and as long as they use the application at least once every seven days (the refresh token lifetime), they stay logged in indefinitely. The fifteen-minute access token expiry happens transparently — the user never sees it because the interceptor handles the refresh automatically.

Email verification is another important piece. When a user registers through the api/v1/auth/register endpoint, their account is created with EmailConfirmed set to false. They cannot log in until they verify their email by clicking a link sent to their inbox. The login handler checks this flag and returns a specific error if the email is not verified, which the frontend displays as a distinct message from invalid credentials. In our development seed data, all demo users have EmailConfirmed set to true so you can log in immediately without needing an email service.

Session management adds another layer. The UserSession entity tracks active login sessions — which device, which IP address, when the session started. Users can view their active sessions and revoke any they do not recognise. Revoking a session invalidates the refresh token associated with it, effectively logging out that device.

Two-factor authentication is supported for admin roles using TOTP — time-based one-time passwords compatible with apps like Google Authenticator or Authy. The infrastructure is provided by ASP.NET Core Identity's built-in 2FA support. When enabled, the login flow requires a second step where the user provides a six-digit code from their authenticator app after entering their password.

Password reset follows a secure flow. The user requests a reset by providing their email. The system generates a time-limited token (using Identity's built-in token provider) and would send it via email. The user clicks the link, provides their new password along with the token, and the system validates the token and updates the password. The endpoint always returns success even if the email does not exist — again, to prevent email enumeration.

The key mental model to carry with you is that the JWT is the source of truth for the request context. It tells us who the user is (sub claim), what role they have (role claim), and which club they belong to (clubId claim). Everything downstream — authorization decisions, tenant resolution, audit logging — flows from those claims. The token is verified on every request by the ASP.NET Core JWT Bearer middleware, which validates the signature, checks the expiration, and populates the HttpContext.User with the claims. By the time your controller action or MediatR handler runs, the user's identity is fully established and trustworthy.
