# Member Management

Member management is the heart of what this platform does for clubs. Every sports club, whether it is a cricket club with fifty members or a football club with three hundred, needs to track who their members are, what status they are in, how to contact them, and what their history looks like. The Members module handles all of this, and it is one of the richest modules in the system in terms of domain complexity.

Let me start with the Member entity itself, because it is one of the most feature-rich entities in the entire codebase. At its core, a member has the basics you would expect — first name, last name, email, phone number, date of birth, and gender. But it goes much deeper than that. A member has a unique member number within their club, formatted with a club prefix like TCC-0001 for Teddington Cricket Club or HUF-0015 for Highbury United FC. This number is generated when the member is created and never changes, even if the member's name or email changes. It serves as a stable human-readable identifier that club administrators can use in conversation, on printed materials, and in correspondence.

Beyond the basics, the Member entity uses EF Core owned types to model structured data that belongs to the member but has its own shape. The Address value object has line one, line two, city, county, postcode, and country. Rather than creating a separate Addresses table with a foreign key back to Members, we store these as columns directly in the Members table — Address_Line1, Address_City, Address_PostCode, and so on. This is a pragmatic choice. An address is not an independent entity with its own lifecycle — it exists only as part of a member. Owned types let us model it as a value object in the domain while keeping the database schema simple and queries fast.

Emergency contacts follow the same pattern. A member can have a primary and secondary emergency contact, each with a name, phone number, and relationship to the member. These are stored as columns in the Members table too — PrimaryEmergency_Name, PrimaryEmergency_Phone, PrimaryEmergency_Relation, and the same for secondary. Medical information is another owned type with conditions, allergies, doctor name, doctor phone, and blood type. All of this lives on the Members table because it is intrinsically part of the member record.

Now, the most important aspect of a member from a domain modelling perspective is their status, and this is modelled as a proper state machine with enforced transitions. A member starts in Pending status when they first register or are added by a club administrator. From Pending, they can only transition to Active — that is when the club approves them, they pay their membership fee, or they complete whatever onboarding the club requires. From Active, they can move to Expired (their membership lapsed and was not renewed), Suspended (the club suspended them for disciplinary reasons or non-payment), or Cancelled (they voluntarily left the club). From Expired or Suspended, they can transition back to Active through reactivation. From Cancelled, there is no way back — that is a terminal state.

This state machine is enforced in the domain entity itself, not in a service layer or a handler. The Activate, Suspend, Expire, Cancel, and Reactivate methods all call a private method called EnsureValidTransition, which checks whether the requested transition is allowed from the current state. If you try to suspend a pending member, you get an InvalidOperationException with a clear message explaining which transitions are valid from the current state. This is domain-driven design in action — the business rules live in the entity where they cannot be bypassed. No matter how the entity is used — from a command handler, from a background job, from a test — the state machine rules are always enforced.

There is also a static method called GetAllowedTransitions that returns the list of valid next states for any given current state. This is useful for the frontend — when displaying a member's profile, the UI can call this to know which action buttons to show. If the member is Active, show Suspend, Expire, and Cancel buttons. If they are Suspended, show only Reactivate.

The MembersDbContext configures all of this with careful attention to constraints and indexes. There is a unique composite index on ClubId plus Email, which means within a single club, no two members can share an email address. But the same person could be a member of multiple clubs with the same email — that is a valid real-world scenario and the index allows it. There is another unique composite index on ClubId plus MemberNumber, ensuring member numbers are unique within a club. There is a non-unique index on ClubId plus Status, which optimizes the very common query of "show me all active members in this club."

The tenant query filter is applied to the Member entity, so all queries automatically filter by the current club. This means a handler that calls dbContext.Members.ToListAsync() will only return members belonging to the authenticated user's club. You never need to manually add a WHERE ClubId equals something — the filter handles it.

Family accounts are a feature that many clubs need. A parent member can have dependents linked to them through the FamilyMember entity. This records the relationship type — parent, spouse, child, sibling — and links two member records together. The primary member is typically the one who pays, and the dependents are covered under a family membership. The IsFamilyAccount flag on the Member entity indicates whether this member is part of a family group, and PrimaryMemberId points to the head of the family if this member is a dependent.

Custom fields provide extensibility without schema changes. A cricket club might want to track "preferred batting position" or "bowling style." A football club might want "preferred foot" or "shirt number." Rather than adding sport-specific columns to the Members table, we have a CustomFieldDefinition entity that lets clubs define their own fields with a name, field type (text, number, date, dropdown), and optional validation rules. The member's custom field values are stored as a JSON string in the CustomFieldValues column. This is a flexible approach that lets each club customize their member data without requiring database migrations.

The module also tracks member status transitions through the MemberStatusTransition entity. Every time a member's status changes, a record is created with the previous status, new status, the timestamp, and who made the change. This gives clubs a complete audit trail of a member's lifecycle — when they joined, when they were activated, if they were ever suspended and why, when they were reactivated. This history is valuable for dispute resolution and for understanding member retention patterns.

Member notes provide a way for club administrators to attach free-text notes to a member's record. These might be disciplinary notes, communication records, special requirements, or anything else that does not fit into the structured fields. Each note has a type (General, Disciplinary, Medical, Financial, Communication), content, and a timestamp with the ID of the user who created it.

The seeder creates seventy-one members across four clubs with a realistic distribution: seventy percent active, ten percent pending, ten percent expired, ten percent suspended. Each has a generated name from a pool of common British first and last names, a unique email, a UK mobile phone number, a date of birth between 1970 and 2010, and a join date within the last three years. The member numbers follow each club's prefix convention. This gives you meaningful data to work with when developing features that display member lists, filter by status, calculate statistics, or test the state machine transitions.

The CSV import feature allows clubs to bulk-import members from spreadsheets, which is essential for onboarding existing clubs that have their member data in Excel. The import process validates each row, creates Member entities, assigns member numbers, and reports any rows that failed validation. This is a common first-time operation when a club joins the platform.

When you are extending the Members module, the key thing to keep in mind is that the Member entity is the aggregate root. All operations on member data go through the Member entity's methods. You do not directly modify properties from outside — you call methods like Activate, Update, SetMemberNumber. This ensures the invariants are always maintained and the state machine cannot be violated.
