# Modular Monolith Architecture

Let me explain why we chose a modular monolith and what that actually means in practice, because it is a deliberate architectural decision that sits between two extremes and understanding the reasoning will help you make good decisions when extending the system. This is not a choice we made by accident or by default — it is the result of carefully weighing the trade-offs for a platform at our stage of development.

On one end of the spectrum you have the traditional monolith — one big project where everything references everything else. The Members service calls the Payments service directly. The Sessions controller imports types from the Events module. Changing one thing can break something completely unrelated because the dependency graph is a tangled web. This is what most projects become over time if you do not actively fight against it. It is easy to build initially but becomes increasingly painful to maintain as the codebase grows.

On the other end you have microservices — separate deployable units that communicate over the network. Each service has its own database, its own deployment pipeline, its own scaling characteristics. This gives you strong isolation and independent deployability, but it introduces enormous operational complexity. You need service discovery, distributed tracing, eventual consistency handling, network failure resilience, API versioning between services, and a team large enough to own and operate each service independently. For a platform with a small team that is still iterating on features, microservices are premature optimization of the worst kind — you pay the complexity cost upfront without yet having the scale that justifies it.

A modular monolith gives you the development-time isolation of microservices with the deployment simplicity of a monolith. You deploy one application, but internally it is structured as independent modules that do not know about each other. You get the benefits of clear boundaries, independent development, and enforced separation without the operational overhead of distributed systems. And critically, if you ever do need to extract a module into a separate service, the boundaries are already clean — it is a straightforward extraction rather than a painful untangling.

In our solution, we have sixteen domain modules. Each module is its own project — its own compiled assembly — with its own domain entities, its own DbContext, its own API controllers, its own MediatR handlers, and its own database schema. The Identity module knows nothing about the Sessions module. The Payments module knows nothing about the Equipment module. They are independent vertical slices of functionality. You can work on the Facilities module for a week without ever opening a file in the Communications module.

The way modules are discovered and registered at startup is through the IModule interface. This is a simple interface in the Shared.Contracts project with three members: a Name property, a RegisterModule method, and a UseModule method. Every module has exactly one class that implements this interface — ClubsModule, IdentityModule, SessionsModule, and so on.

At startup, Program.cs performs assembly scanning. It looks at all loaded assemblies whose names start with "TheLeague.Modules," finds every type that implements IModule, instantiates each one, and calls RegisterModule. This is how each module registers its DbContext, its services, its MediatR handlers, and anything else it needs with the dependency injection container. The Host project references all module projects (so their assemblies are loaded), but it does not know anything about their internals. It just calls the IModule interface methods.

The UseModule method is called after the application is built, during the middleware configuration phase. This allows modules to register their own middleware if needed — though most modules do not need any. The pattern is there for cases where a module needs to intercept requests in a module-specific way.

Each module follows Clean Architecture internally. This is not just a suggestion — it is enforced by architecture tests in the TheLeague.Tests.Architecture project using NetArchTest. The Domain folder contains entities and value objects — pure business logic with no infrastructure dependencies. Domain classes do not reference EF Core, do not reference ASP.NET Core, do not reference MediatR. They are plain C# classes with business methods. The Application folder contains commands, queries, handlers, DTOs, and validators — the use cases of the module. Application depends on Domain but not on Infrastructure. The Infrastructure folder contains the DbContext, any repository implementations, and external service integrations. Infrastructure depends on both Application and Domain. The Api folder contains the controllers — the HTTP entry points. Api depends on Application.

Dependencies flow inward: Api references Application, Application references Domain, Infrastructure references Application and Domain. Nothing references Api. Nothing in Domain references anything outside Domain. This ensures that your business logic is never contaminated by infrastructure concerns. You can change your database from SQL Server to PostgreSQL without touching a single domain entity. You can swap your HTTP framework without modifying your command handlers.

The modules communicate with each other through integration events, not direct references. This is a critical rule. If the Payments module needs to notify the Members module that a payment was received, it does not call a method on a Members service. Instead, it publishes an integration event — PaymentCompletedEvent — through the IIntegrationEventBus. The Members module subscribes to that event and handles it independently. The Payments module does not know or care what happens when a payment is completed. It just announces the fact.

The integration event bus is currently an in-process implementation — events are dispatched synchronously within the same application process. This is fine for a monolith. If we ever extract modules into separate services, the event bus implementation would change to use a message broker like RabbitMQ or Azure Service Bus, but the module code would not change at all. The modules publish and subscribe to events through the same IIntegrationEventBus interface regardless of the underlying transport.

The shared layer — TheLeague.Shared — provides the common infrastructure that all modules need. It is split into three projects. Shared.Domain has the base entity classes (BaseEntity, TenantEntity, AuditableEntity), value objects (Address, EmergencyContact, MedicalInfo), and the thirty-four enums used across the system. Shared.Contracts has the IModule interface, integration event definitions, and service interfaces like ITenantService and ICacheService. Shared.Infrastructure has the concrete implementations — the tenant service, Redis caching, the MediatR pipeline behaviours, the middleware, and the authorization infrastructure. Every module references Shared.Contracts and Shared.Domain. Only the Host project references Shared.Infrastructure directly.

The database strategy reinforces the module separation. All modules share a single SQL Server database, but each module has its own schema. The Identity module uses the "identity" schema, Clubs uses "clubs," Members uses "members," Sessions uses "sessions," and so on. Each module's DbContext is configured with HasDefaultSchema in OnModelCreating. Each module's migration history is stored in its own schema's __EFMigrationsHistory table. This means you can run migrations for one module without affecting others. You can look at the database in SSMS and immediately see which tables belong to which module by their schema prefix.

The practical benefit for you as a developer is that when you are working on a feature, you only need to understand one module. You open the Sessions module, you see its entities, its handlers, its controller, and its DbContext. Everything you need is right there in one project. You do not need to trace through a web of cross-cutting dependencies to understand how session booking works. The module is self-contained.

When you need to add a completely new feature area — say a Sponsorship module — you create a new project, implement IModule, define your entities and DbContext, write your handlers and controller, and add a project reference from the Host. The module is automatically discovered at startup. You do not need to modify Program.cs or any other module. That is the power of the modular monolith — new features are additive, not invasive.
