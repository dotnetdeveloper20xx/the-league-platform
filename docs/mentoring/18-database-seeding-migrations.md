# Database Seeding and Migrations

Let me explain how we manage the database lifecycle in this project — from empty database to fully populated development environment — because understanding this process is essential for your day-to-day work. You will need to create migrations when you change entities, you will need to understand how seeding works when you add new modules, and you will need to know how to reset your local database when things go wrong. This is the operational knowledge that makes you productive.

EF Core migrations are the mechanism by which we evolve the database schema over time. When you add a new property to an entity, create a new entity, change a relationship, or modify an index, you need to create a migration that captures those changes as executable SQL. The migration is a C# class with an Up method (apply the change) and a Down method (reverse the change). When the application starts, the MigrationRunner applies any pending migrations — migrations that exist in code but have not yet been applied to the database.

In our modular monolith, migrations are per-module. Each module has its own DbContext with its own schema, and each module's migrations are stored in its own project under Infrastructure/Persistence/Migrations. When you modify the Members module's entities, you generate a migration in the Members module. When you modify the Sessions module's entities, you generate a migration in the Sessions module. They are completely independent — you can add a migration to one module without affecting any other module's migration history.

The command to generate a migration follows a specific pattern because of our multi-project structure. You run it from the solution root directory, specifying the project that contains the DbContext (the module), the startup project (the Host, which has the connection string and DI configuration), the specific DbContext class name (because the Host references all sixteen DbContexts), and the output directory within the module. The migration name should be descriptive — not "Initial" or "Update" but something like "AddMemberNotesTable" or "AddSessionWaitlistSupport" that tells you what changed without opening the file.

The startup project parameter is critical and often confuses people new to multi-project EF Core setups. The EF Core tools need to build and run the startup project to get the DbContext configuration — the connection string, the model configuration, the service registrations. The module project alone does not have enough context to configure the DbContext because the connection string lives in appsettings.json in the Host project. So you always specify the Host as the startup project, even though the migration files end up in the module project.

The MigrationRunner class in the Host project is responsible for applying migrations at startup. It iterates through all sixteen DbContext types and calls Database.MigrateAsync() on each one. MigrateAsync is idempotent — if all migrations have already been applied, it does nothing. If there are pending migrations, it applies them in order. This means you can always start the application and trust that the database schema will be up to date. You do not need to manually run update-database commands during development (though we have a script for that too, for cases where you want to apply migrations without starting the full application).

Each module's migrations are tracked independently in the database. The Identity module's migration history is in the identity.__EFMigrationsHistory table. The Clubs module's history is in clubs.__EFMigrationsHistory. And so on for all sixteen modules. This per-schema tracking means that applying a migration for one module does not interfere with another module's migration state. You could roll back the Members module's last migration without affecting any other module.

Now let me talk about seeding, because this is where the development experience really comes together. The DatabaseSeeder class in the Host project populates the database with realistic demo data on first startup. It runs after migrations (so the tables exist) and is idempotent (it checks for existing data before inserting, so running it twice does not create duplicates). The seeder is only active in the Development environment — it does not run in production.

The seeding order matters because of foreign key relationships. You cannot create a Member without a Club existing first (the Member has a ClubId foreign key). You cannot create a Membership without both a Member and a MembershipType existing. You cannot create a Payment without a Member existing. The seeder respects this dependency order: Identity (roles and users) first, then Clubs, then Members, then MembershipTypes, then Memberships, then Facilities, Equipment, Sessions, Events, Competitions, and finally Payments.

The SeedData class contains all the demo data definitions. It uses deterministic GUIDs — hardcoded GUID values rather than randomly generated ones. This is important for two reasons. First, it makes seeding idempotent — the seeder can check "does a club with this specific ID already exist?" and skip it if so. Second, it allows cross-module references — the Members seed data references Club IDs that are defined in the Clubs seed data, and the Payments seed data references Member IDs defined in the Members seed data. If IDs were random, these cross-references would be impossible.

The deterministic GUID pattern uses a structured format where the first segment indicates the entity type (40 for members, 50 for membership types, 70 for facilities, 80 for sessions, etc.) and subsequent segments encode the club and sequence number. This makes it easy to identify what kind of entity a GUID represents just by looking at it — useful when debugging database queries.

The seeder uses each module's specific DbContext rather than a generic DbContext. This is important because each module's DbContext has its own entity configurations, owned type mappings, and query filters. Using the wrong context would result in entities being saved without proper configuration, potentially missing columns or violating constraints. The ClubsDbContext knows how to persist Club entities. The MembersDbContext knows how to persist Member entities with their owned Address and EmergencyContact types. Each seeder method resolves the correct DbContext from the DI container.

For entities with rich domain logic — like Members with their state machine — the seeder uses the domain methods rather than directly setting properties. It calls Member.Create() to get a new member in Pending status, then calls Activate() to transition to Active, or Activate() followed by Suspend() to get to Suspended. This ensures the domain invariants are respected even during seeding. For properties that are not exposed through public methods (like setting a specific JoinedDate for historical realism), the seeder uses reflection to set private properties. This is acceptable in seed code but would never be done in production handler code.

The seed data is comprehensive and realistic. Four clubs representing different sports (two cricket, one football, one hockey). Twelve users across different roles (one SuperAdmin, eight ClubManagers, three Coaches). Seventy-one members with realistic names, contact details, and a distribution of statuses. Fifteen membership types with sport-appropriate pricing. Sixteen facilities with realistic capacities. Nineteen equipment items with values and depreciation rates. Three hundred and sixty-five sessions spanning six months of history. Fourteen upcoming events. Four competitions with teams, matches, and calculated standings. Two hundred and fifty-one payments spanning twelve months. This gives you a rich, realistic dataset that makes development and demo scenarios feel authentic.

When you need to reset your local database — because migrations got out of sync, because you want fresh seed data, or because you changed entity structures significantly — the process is straightforward. Drop the database (either through SSMS or using the sqlcmd command), then start the application. The MigrationRunner will recreate all tables from the migration files, and the DatabaseSeeder will populate them with fresh demo data. The whole process takes about ten seconds. We also have batch scripts (generate-migrations.bat and update-database.bat) for when you need to regenerate or apply migrations outside of the application startup flow.

One important consideration when adding new seed data is maintaining referential integrity across modules. If you add a new entity that references Members (like a new type of booking or assessment), your seed data needs to use Member IDs that actually exist in the Members seed data. The deterministic GUID pattern makes this possible — you can reference SeedData.Members[0].Id knowing exactly which member that is. But you need to be careful about the seeding order — your new seeder method must run after the Members seeder.

The warnings you see at startup about optional dependents using table sharing (EmergencyContact, Address, MedicalInfo) are EF Core informational warnings, not errors. They are telling you that because these owned types have all-nullable properties, EF Core cannot distinguish between "the member has an address where all fields happen to be null" and "the member has no address at all." In practice this does not matter for our use case — if all address fields are null, we treat it as no address regardless. The warnings can be suppressed in the DbContext configuration if they bother you, but they are harmless.
