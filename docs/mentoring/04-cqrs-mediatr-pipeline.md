# CQRS and the MediatR Pipeline

This is probably the most important architectural pattern to understand in the entire codebase, because every single operation flows through it. Whether a user is logging in, booking a session, creating a club, or marking attendance, the request travels through the same pipeline with the same cross-cutting behaviours applied consistently. Once you understand this pattern, you understand how to build anything in this system.

CQRS stands for Command Query Responsibility Segregation. The core idea is deceptively simple: separate the code that changes data from the code that reads data. Commands mutate state — they create, update, or delete things. Queries read state — they fetch and return data without changing anything. In our codebase, these are literally different classes. A CreateClubCommand is a command. A GetClubByIdQuery is a query. They have different handlers, they can have different validation rules, and they can be optimized independently. You could, for example, have queries read from a denormalized read model or a cache, while commands always go through the full domain logic. We do not do that level of separation yet, but the architecture supports it if we ever need it.

We implement CQRS using MediatR, which is a mediator pattern library for dotnet. The mediator pattern decouples the sender of a request from its handler. Instead of a controller directly calling a service method, the controller creates a request object (a command or query) and sends it to MediatR. MediatR looks at the type of the request, finds the registered handler for that type, and invokes it. The controller never knows which class handles the request — it just knows that someone will handle it and return a result.

This decoupling has several benefits. First, your controllers become extremely thin. They do three things: receive the HTTP request, create a command or query, and return the result. There is no business logic in controllers. Second, your handlers are focused and testable. Each handler does one thing — handle one specific command or query. You can unit test it in isolation by constructing the command, calling the handler, and asserting the result. Third, and most importantly, it gives us a pipeline.

The MediatR pipeline is a series of behaviours that run before and after every handler. Think of it like middleware for your business logic. Just as ASP.NET Core middleware wraps around your HTTP request processing, MediatR behaviours wrap around your command and query handling. The behaviours are registered in a specific order, and that order matters because each behaviour can short-circuit the pipeline or modify the request before passing it to the next behaviour.

Let me walk you through our pipeline in the exact order the behaviours execute. First is LoggingBehaviour. Every command and query that enters the system gets logged with its type name, a correlation ID (which ties it back to the HTTP request), and a timestamp. When the handler completes, the behaviour logs the duration. This is invaluable for debugging in production — you can trace a request from the HTTP layer through the pipeline to the handler and back, seeing exactly how long each step took. If a user reports that "creating a session was slow," you can search the logs by correlation ID and see that the SessionsDbContext.SaveChangesAsync call took three seconds because of a database lock.

Next is ValidationBehaviour. This is where FluentValidation kicks in. For every command that enters the pipeline, the behaviour checks whether there are any registered IValidator implementations for that command type. If there are, it runs all validation rules. If any rule fails, the behaviour short-circuits the pipeline immediately — the handler never executes. Instead, a validation failure result is returned directly to the controller. This is a massive win for code cleanliness. Your handlers can assume their input is valid. You never need to write defensive null checks or range validations inside a handler. All of that lives in the validator class, which is a separate, focused, testable unit.

The validator classes use FluentValidation's fluent API to define rules. A CreateClubCommand validator might specify that the Name property must not be empty and must be less than two hundred characters, that the Slug must match a URL-friendly pattern, and that the ContactEmail must be a valid email format. These rules are declarative and readable. When you need to add a new validation rule, you add it to the validator. When you need to understand what validation a command has, you look at its validator. There is no hunting through handler code for scattered if-statements.

Then comes TenantBehaviour. This ensures the tenant context is properly established before any handler runs. For commands and queries that operate on tenant-scoped data, this behaviour verifies that the ITenantService has a valid CurrentTenantId. If a tenant-scoped operation is attempted without a tenant context — which would indicate a bug in the middleware or a misconfigured background job — the behaviour can log a warning or throw an exception. It is a safety net that catches configuration errors before they result in data leaks or empty results.

PerformanceBehaviour is the fourth in the pipeline. It starts a stopwatch before the handler executes and stops it after. If the handler takes longer than a configurable threshold — typically five hundred milliseconds — it logs a performance warning with the command type, the duration, and the correlation ID. This is your early warning system for performance regressions. If someone introduces a new query that does a full table scan, the performance behaviour will start logging warnings in development and staging before it ever hits production. You can set up alerts on these log entries to catch problems proactively.

Finally, TransactionBehaviour wraps command handlers in a database transaction. When a command handler starts, the behaviour begins a transaction. If the handler completes successfully, the transaction is committed. If the handler throws an exception, the transaction is rolled back. This ensures atomicity — either all changes from a command are persisted, or none of them are. If a command creates a member and then fails while creating their initial membership, the member creation is rolled back too. You never end up with half-completed operations in the database.

Note that TransactionBehaviour only applies to commands, not queries. Queries do not modify data, so they do not need transaction wrapping. The behaviour checks whether the request is a command (implements ICommand or similar marker) before starting a transaction.

The beauty of this pipeline is that it is completely transparent to the handler code. When you write a new command handler, you do not think about logging, validation, tenancy, performance monitoring, or transactions. You just write the business logic. The pipeline handles all the cross-cutting concerns automatically. This is the open-closed principle in action — the pipeline is open for extension (you can add new behaviours) but closed for modification (existing handlers do not need to change when you add a new behaviour).

From a practical standpoint, when you need to add a new operation to the system, the process is always the same. You create a command or query record — a simple immutable data class that carries the request parameters. You create a handler class that implements IRequestHandler with the command as the request type and the result as the response type. Optionally, you create a validator class that implements IValidator for the command type. You register nothing manually — MediatR scans the module assemblies at startup and discovers all handlers and validators automatically through assembly scanning.

The controller action is trivially simple. It receives the HTTP request, maps it to a command (often the command IS the request body, deserialized directly), calls mediator.Send with the command, and returns the result. If the result indicates success, return 200 or 201. If it indicates failure, return 400 or 404. That is the entire controller pattern, and it is consistent across all sixteen modules and all hundred-plus endpoints.

One more thing worth mentioning is the Result pattern we use for handler responses. Instead of throwing exceptions for business logic failures (like "member not found" or "session is full"), handlers return a Result object that indicates success or failure with an error message. This keeps the pipeline clean — exceptions are reserved for truly exceptional situations (database connection lost, null reference bugs), while expected business failures flow through the normal return path. The TransactionBehaviour only rolls back on exceptions, not on business failure results, which is the correct behaviour — if a handler returns "session is full," there is nothing to roll back.
