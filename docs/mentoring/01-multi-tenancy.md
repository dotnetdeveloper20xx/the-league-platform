# Multi-Tenancy

So let me tell you about how we handle multi-tenancy in this platform, because it is one of those foundational decisions that touches absolutely everything else you will ever build on top of this codebase. When I say multi-tenant, what I mean is that a single deployed instance of this application serves multiple sports clubs simultaneously, and each club's data is completely isolated from every other club's data. A member of Teddington Cricket Club should never, under any circumstances, see data belonging to Richmond Hockey Club. That is the contract we are upholding, and it is non-negotiable.

Now, there are several ways to achieve multi-tenancy in a software system, and each comes with its own trade-offs. The first approach is database-per-tenant, where every club gets its own physical database. This gives you the strongest possible isolation — if one club's database gets corrupted or needs to be restored from backup, it doesn't affect anyone else. The downside is operational complexity. When you have five hundred clubs, you have five hundred databases to manage, migrate, back up, and monitor. Connection string management becomes a nightmare, and cross-tenant reporting requires querying across hundreds of databases.

The second approach is schema-per-tenant within a shared database. Each club gets its own set of tables in a dedicated schema. This is better operationally but still creates a lot of database objects and makes migrations more complex — you need to apply schema changes to every tenant's schema independently.

The third approach, and the one we use, is shared database with a discriminator column. Every tenant-scoped table has a ClubId column, and we use EF Core's global query filters to ensure that every query automatically includes a WHERE clause filtering by the current tenant's ClubId. This means you literally cannot forget to filter by tenant — the framework does it for you. The trade-off is that isolation is logical rather than physical, so a bug in the query filter logic could theoretically expose cross-tenant data. That is why we have multiple layers of defence, which I will explain.

The way this works at runtime starts with a piece of middleware called TenantMiddleware. This middleware sits in the ASP.NET Core request pipeline and runs on every incoming HTTP request. Its job is to figure out which club this request belongs to and set that context for the duration of the request. The first thing it does is check whether the user is authenticated. If they are, it looks for a claim called clubId in their JWT token. Remember, when we issue a JWT during login, we embed the user's club ID as a claim. So for a club manager at Teddington Cricket Club, their token contains a clubId claim with Teddington's GUID.

If the middleware finds that claim, it calls SetCurrentTenant on our ITenantService with that GUID. ITenantService is a scoped service, which is a critical detail. Scoped means a new instance is created for each HTTP request and disposed when the request ends. So the tenant context is inherently request-scoped — it cannot leak between concurrent requests. Two requests from different clubs being processed simultaneously on different threads each have their own ITenantService instance with their own tenant ID.

There is also a fallback mechanism for platform-level operations. A SuperAdmin is not tied to any specific club — they manage the platform itself. For scenarios where a SuperAdmin needs to view or modify a specific club's data, the middleware checks for an X-Tenant-Id header. This allows admin tools to explicitly specify which club's context they want to operate in. If neither the JWT claim nor the header provides a tenant ID, the service simply has no tenant set, and queries against tenant-scoped entities will return nothing. That is a safe default — it is better to return no data than to accidentally return all data.

Now let me explain how the query filters actually work in EF Core. In each module's DbContext, during OnModelCreating, we configure a query filter on every tenant-scoped entity. The filter looks something like HasQueryFilter with a lambda that checks whether the entity's ClubId equals the current tenant ID from the tenant service. EF Core applies this filter to every LINQ query against that entity — whether it is a simple Find by ID, a complex join, or a projection. You do not need to remember to add the filter yourself. It is always there.

The entity hierarchy supports this cleanly. We have BaseEntity at the root, which gives you Id (a GUID), CreatedAt, and UpdatedAt. Then TenantEntity extends that with a ClubId property. Then AuditableEntity adds CreatedBy and UpdatedBy on top of TenantEntity. Most domain entities in the system inherit from TenantEntity or AuditableEntity, which means they all carry that ClubId discriminator. When you create a new entity for a new feature, you inherit from TenantEntity and the multi-tenancy support comes for free.

One thing that is worth understanding deeply is how this interacts with the MediatR pipeline. We have a TenantBehaviour that runs as part of the pipeline for every command and query. Its job is to ensure that the tenant context is properly set before any handler executes. This is a defence-in-depth approach — even if somehow the middleware did not set the tenant (maybe a misconfigured test, maybe a background job that forgot to set context), the pipeline behaviour acts as a second checkpoint. It can log a warning or throw an exception if a tenant-scoped operation is attempted without a tenant context.

There are some entities that are intentionally not tenant-scoped. The Club entity itself, for example, does not have a ClubId discriminator — it IS the tenant. Subscription tier configurations are platform-wide, not per-club. User accounts in the Identity module are not strictly tenant-scoped either, because a SuperAdmin exists outside any club context. When you encounter an entity that inherits from BaseEntity rather than TenantEntity, that is a deliberate design choice indicating it is either a platform-level entity or the tenant root itself.

Testing with multi-tenancy requires some awareness. In integration tests, you need to set up the tenant context before running queries, otherwise the query filters will return empty results. The IgnoreQueryFilters method on EF Core queries bypasses the filter — this is useful in tests and in specific admin scenarios where you genuinely need to see all data regardless of tenant. But in production handler code, you should never use IgnoreQueryFilters unless you have a very specific reason and have documented why.

The practical implication for you as a developer joining this team is straightforward. When you create a new entity, inherit from TenantEntity. When you configure it in your DbContext's OnModelCreating, add a query filter using the tenant service's CurrentTenantId. When you create seed data, make sure every record has a valid ClubId that corresponds to one of the seeded clubs. When you write a new command handler, trust that the tenant context is already set by the time your handler runs — the middleware and pipeline behaviour have already done their job. And when you write tests, remember to either set up the tenant context or use IgnoreQueryFilters for verification queries.

The beauty of this approach is that once you understand the pattern, you never think about tenant isolation again in your day-to-day feature work. You write code as if you are building for a single club, and the infrastructure ensures that code automatically works correctly in a multi-tenant environment. That is the power of a well-implemented cross-cutting concern — it becomes invisible to the developers building features on top of it.
