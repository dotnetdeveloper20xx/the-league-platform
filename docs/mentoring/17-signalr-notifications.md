# SignalR and Real-Time Notifications

Let me explain how real-time communication works in this platform, because it is one of those features that transforms the user experience from "checking a website" to "being connected to your club." When a goal is scored in a match, members watching from home should see the score update instantly — not after they refresh the page. When a session is cancelled, members who have booked should get a notification immediately — not whenever they next happen to check their email. When a waitlist spot opens up, the next person in line should be notified in real time so they can grab it before someone else does. All of this is powered by SignalR.

SignalR is Microsoft's library for adding real-time web functionality to applications. Under the hood, it uses WebSockets when available (which is most of the time in modern browsers), falling back to Server-Sent Events or long polling when WebSockets are not supported. The beauty of SignalR is that you write your code against a high-level abstraction (hubs and clients) and the library handles the transport negotiation, connection management, reconnection logic, and message serialization for you. You do not need to think about WebSocket frames or HTTP streaming — you just call methods on connected clients.

This platform has two SignalR hubs: NotificationHub and MatchCentreHub. Each serves a different purpose and has different connection patterns, authentication requirements, and message types. Let me walk through each one.

The NotificationHub is the general-purpose real-time notification channel. Every authenticated user connects to this hub when they load the application. It handles notifications like "your booking has been confirmed," "your payment was successful," "a new event has been published," "your waitlist position has been promoted," and "training tomorrow has been cancelled." These are the same notifications that might also be sent by email or SMS, but the real-time channel provides instant feedback when the user is actively using the platform.

When a user connects to the NotificationHub, they are added to a group based on their user ID. This means the server can send notifications to a specific user by targeting their group. The notification service (a backend service that other modules call when they want to notify a user) checks whether the user is currently connected. If they are, it sends the notification through SignalR for instant delivery. Regardless of connection status, it also persists the notification to the database so the user can see it later in their notification history. This dual-delivery approach ensures notifications are never lost — they arrive instantly if the user is online, and they are waiting in the notification tray when the user next logs in if they were offline.

The MatchCentreHub is specialised for live scoring during matches. It uses a different connection pattern — users join and leave match-specific groups as they navigate to and away from the match centre page. When a user opens the live score page for a specific match, the client calls a JoinMatch method on the hub, passing the match ID. The hub adds the user's connection to a group named after that match ID. When they navigate away, the client calls LeaveMatch, and the connection is removed from the group. This means score updates are only broadcast to users who are actively watching that specific match, which is efficient — you are not pushing cricket scores to someone who is looking at the football fixtures.

The MatchCentreHub handles several types of real-time messages. ScoreUpdate is the most common — it carries the new home score, away score, and a timestamp. MatchEvent carries details of specific events like goals, wickets, cards, or substitutions — including the time, the player involved, and a description. StatusChange indicates that the match status has changed (started, half-time, resumed, ended). Commentary carries text updates from the scorer or commentator. Each message type has a corresponding method that the server calls on connected clients, and the Angular frontend handles each type differently in the UI.

Authentication for SignalR connections requires special handling because WebSockets do not support custom HTTP headers in the same way that regular HTTP requests do. You cannot send an Authorization header with a WebSocket upgrade request in all browsers. The standard solution, which we use, is to pass the JWT token as a query string parameter during the connection handshake. The SignalR configuration in Program.cs includes an event handler for OnMessageReceived that checks for a token in the query string (typically as an "access_token" parameter) and sets it as the authentication token for the connection. This means the WebSocket connection is authenticated with the same JWT that the user obtained during login, and all the same claims (user ID, club ID, roles) are available on the hub context.

The security implications of query string tokens are worth understanding. Unlike headers, query string parameters can appear in server logs, browser history, and proxy logs. This is mitigated by several factors: the token is short-lived (our JWTs expire after fifteen minutes), the connection is over HTTPS (so the query string is encrypted in transit), and server-side logging is configured to exclude query strings from access logs. It is not ideal from a security purist's perspective, but it is the standard approach for WebSocket authentication and the trade-offs are well understood and acceptable.

On the frontend, the Angular application uses the @microsoft/signalr npm package to connect to the hubs. The SignalR service (a singleton Angular service) manages the connection lifecycle. When the user logs in, the service establishes a connection to the NotificationHub using the user's JWT token. It registers handlers for incoming messages (notification received, notification count updated) and exposes observables that components can subscribe to. When the user logs out, the connection is gracefully closed.

The connection includes automatic reconnection logic. If the WebSocket connection drops (network interruption, server restart, mobile device going to sleep), the SignalR client automatically attempts to reconnect with exponential backoff. During the disconnection period, the UI shows an offline indicator — a subtle banner or icon that tells the user they are temporarily disconnected from real-time updates. When the connection is re-established, the indicator disappears and any missed notifications are fetched from the API to ensure nothing was lost during the gap.

The offline indicator is a small but important UX detail. Without it, a user might think the system is working normally when in fact they are not receiving real-time updates. They might be watching a live match score that has not updated for five minutes, not realising that their connection dropped. The offline indicator makes the connection state visible, setting appropriate expectations and prompting the user to check their network if the disconnection persists.

The notification service on the backend is the bridge between domain events and real-time delivery. When any module wants to notify a user, it calls the notification service with the recipient's user ID, the notification type, a title, a message body, and optionally a link (a URL within the application that the notification relates to). The service persists the notification to the database, increments the user's unread count, and then attempts real-time delivery through the NotificationHub. If the user is connected, they see the notification appear instantly in their notification dropdown. If not, it will be there waiting when they next open the app.

The notification types are categorised to allow filtering and prioritisation in the UI. Types include BookingConfirmed, PaymentReceived, PaymentFailed, EventPublished, SessionCancelled, WaitlistPromoted, MembershipExpiring, and MatchScoreUpdate. Each type has an associated icon and colour in the frontend, making it easy for users to scan their notifications and identify what needs attention. High-priority notifications (like payment failures or session cancellations) can be configured to also trigger a browser notification (using the Web Notifications API) even if the user is on a different tab.

The Angular app's connection to the hubs follows a specific lifecycle. The SignalR service is initialised in the app's root module. On application startup, if the user is already authenticated (they have a valid token in local storage), the service immediately connects to the NotificationHub. If the user is not authenticated, the service waits. When the authentication service emits a login event, the SignalR service connects. When it emits a logout event, the SignalR service disconnects. This ensures that the WebSocket connection exists only when there is an authenticated user, and that the connection uses the current valid token.

For the MatchCentreHub, the connection is lazier. It is only established when the user navigates to a page that needs live scoring data. The match centre component, on initialisation, tells the SignalR service to connect to the MatchCentreHub and join the specific match group. On destruction (when the user navigates away), it leaves the group and optionally disconnects from the hub entirely. This keeps resource usage minimal — there is no point maintaining a MatchCentreHub connection for a user who is looking at the membership list.

The scalability considerations for SignalR in a multi-tenant platform are worth mentioning. Each connected user maintains a persistent connection to the server. With thousands of clubs and potentially tens of thousands of concurrent users, that is a lot of connections. In a single-server deployment, this is fine — modern servers can handle hundreds of thousands of concurrent WebSocket connections. But in a scaled-out deployment with multiple server instances behind a load balancer, you need a backplane — a shared message bus that ensures a message sent from one server instance reaches clients connected to other instances. Azure SignalR Service or Redis can serve as this backplane. For our development environment, a single server is sufficient, but the architecture is ready for scale-out when needed.

One design decision worth discussing is why we have two separate hubs rather than a single hub that handles everything. The answer is separation of concerns and connection lifecycle. The NotificationHub is always connected for authenticated users — it is a persistent background channel. The MatchCentreHub is connected only when needed — it is a temporary foreground channel. Combining them would mean either always maintaining the match centre connection (wasteful) or dynamically subscribing and unsubscribing within a single hub (complex). Separate hubs keep the responsibilities clear and the connection management simple.

When you are building features that need real-time updates, the pattern is consistent: your domain logic does its work, publishes an integration event or calls the notification service, and the real-time infrastructure handles delivery. You do not need to think about WebSockets or connection management in your feature code. You just say "notify user X that Y happened" and the infrastructure takes care of the rest. That separation of concerns is what makes the real-time features maintainable as the platform grows.

The notification persistence model ensures that no notification is ever lost. Every notification sent through the NotificationHub is also written to a Notifications table in the database. This table stores the recipient user ID, the notification type, title, body, link, read status, and created timestamp. When a user opens their notification dropdown, the UI fetches unread notifications from the API (not from SignalR — SignalR is for real-time push, the API is for historical retrieval). When the user clicks a notification, it is marked as read. The unread count badge in the header is updated both by SignalR (when new notifications arrive) and by the API (when the page loads and needs the current count).

The error handling and resilience patterns for SignalR connections are important for a good user experience. Network connections are unreliable — they drop when users go through tunnels, switch between WiFi and mobile data, or when their device goes to sleep. The @microsoft/signalr client handles reconnection automatically with configurable retry policies. Our configuration uses an exponential backoff starting at zero seconds (immediate retry), then two seconds, then ten seconds, then thirty seconds, with a maximum of five retry attempts before giving up. If all retries fail, the offline indicator is shown and the user can manually trigger a reconnection by clicking it.

The message ordering guarantee is another consideration. SignalR guarantees that messages sent from the server to a specific client arrive in order. But messages sent to a group (like score updates for a match) might arrive at different clients at slightly different times due to network latency. This is generally acceptable for our use cases — if one user sees a goal notification half a second before another user, that is fine. But for the score display itself, we use the timestamp on each update to ensure the UI always shows the latest score, even if messages arrive out of order. If a score update arrives with a timestamp older than the currently displayed score, it is ignored. This prevents the rare but possible scenario where a delayed message causes the score to briefly show an outdated value.
