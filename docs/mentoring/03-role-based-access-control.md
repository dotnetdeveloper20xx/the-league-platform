# Role-Based Access Control

Alright, let me talk you through how we control who can do what in this system, because authorization is one of those things that seems simple until you start thinking about the edge cases. We have five roles, and they form a clear hierarchy of access. Understanding how they work together with the tenant system gives you the complete picture of our security model.

SuperAdmin sits at the top of the hierarchy. This is the platform owner — the person or team that operates The League as a SaaS business. A SuperAdmin can see all clubs, manage subscriptions, view platform-wide analytics, and perform operations that span multiple tenants. They are not tied to any specific club, which means their JWT token does not contain a clubId claim. When a SuperAdmin needs to view a specific club's data, they use the X-Tenant-Id header to specify which club's context they want to operate in.

ClubManager is the next level down. This is the person who runs a specific club — the chairman, the secretary, the administrator. They have full control over their own club's operations: managing members, creating sessions, setting up events, configuring membership types, viewing financial reports, and managing staff. But they can only see and modify data within their own club. Their JWT contains a clubId claim that locks them into their club's tenant context.

Coach has limited management capabilities focused on their area of responsibility. They can view member lists, manage session attendance, mark scores in competitions, and access training-related features. They cannot manage finances, modify club settings, or perform administrative operations. A coach is tied to a specific club through their clubId claim.

Staff is similar to Coach but oriented toward administrative tasks rather than sporting ones. They might handle front-desk operations, process walk-in payments, or manage facility bookings. The exact permissions can be customized per club, but the role provides a baseline set of capabilities that is less than ClubManager but more than Member.

Member is the most restricted role. Members can view their own profile, book sessions, register for events, view their payment history, manage their family account, and update their preferences. They cannot see other members' data (beyond what is publicly visible like team rosters), cannot modify club settings, and cannot perform any administrative operations.

The way we enforce this is through a custom authorization system built on top of ASP.NET Core's policy-based authorization infrastructure. We have a custom attribute called RequireRoleAttribute that you will see on controller actions throughout the codebase. When you decorate an action with RequireRole and pass in one or more role names, it means only users with one of those roles can access that endpoint. If a user with the Member role tries to hit an endpoint that requires ClubManager, they get a 403 Forbidden response.

Under the hood, this works through two components. The first is PermissionPolicyProvider, which implements IAuthorizationPolicyProvider. When ASP.NET Core encounters a RequireRole attribute during request processing, it asks the policy provider for a policy matching the attribute's configuration. Our provider dynamically creates an authorization policy that requires the user to have a role claim matching one of the specified roles. This dynamic creation means we do not need to pre-register every possible combination of roles as named policies.

The second component is PermissionAuthorizationHandler, which implements IAuthorizationHandler. This is the class that actually evaluates whether the current user satisfies the policy's requirements. It reads the user's role claim from the JWT and checks whether it matches any of the allowed roles specified in the policy. If it matches, authorization succeeds. If not, it fails and the framework returns a 403.

Now here is something important about how roles interact with tenancy, because these two systems work together as complementary layers of security. Roles control what actions you can perform — can you create a session? Can you view financial reports? Can you manage members? Tenancy controls what data you can see — you can only see data belonging to your club. These are orthogonal concerns that combine to form the complete security model.

Consider this scenario: a ClubManager at Teddington Cricket Club somehow crafts a request to view a member that belongs to Richmond Hockey Club. Even though they have the ClubManager role (which grants permission to view members), the tenant query filter ensures that only members with ClubId matching Teddington's ID are returned. The Richmond member simply does not exist from Teddington's perspective. The role says "you are allowed to view members" and the tenant filter says "but only your club's members." Both checks must pass for data to be returned.

In practice, when you are building a new endpoint, you think about authorization like this. First, ask yourself: who should be able to call this endpoint? If it is a club management operation like creating a session or updating club settings, you put RequireRole with Roles.ClubManager. If it is something a member should also be able to do, like viewing their own bookings or registering for an event, you add Roles.Member to the list. If it is a platform-wide operation like viewing all clubs or managing subscription tiers, it is RequireRole with Roles.SuperAdmin only.

One pattern you will notice throughout the codebase is that some endpoints are accessible to multiple roles but return different data depending on the role. For example, a ClubManager calling the members list endpoint gets all members in their club with full details. A Member calling a similar endpoint might only get their own profile, or a limited view of other members (just names for a team roster, for instance). This differentiation happens in the MediatR handler, not at the authorization level. The authorization gate just says "you are allowed through this door" and the handler decides what you get to see once you are inside.

The Roles class itself is deliberately simple. It is a static class with string constants for each role name and an All array that contains all five. The All array is used during database seeding to ensure all roles exist in the ASP.NET Core Identity role store before any users are created. When a user is created during seeding or registration, they are assigned exactly one role through UserManager.AddToRoleAsync.

The system is designed to be extended without breaking existing functionality. If you needed to add a new role — say Treasurer for someone who only handles financial operations — you would add the constant to the Roles class, add it to the All array so it gets seeded, and then start using it in RequireRole attributes on the relevant endpoints. Existing endpoints that do not include the new role in their RequireRole list will automatically deny access to users with that role. You do not need to modify existing authorization logic.

One thing to be aware of is that our current implementation uses role-based authorization rather than permission-based authorization. The difference is that with roles, you check "is this user a ClubManager?" whereas with permissions, you would check "does this user have the manage-sessions permission?" Permission-based systems are more granular — you could give a Coach the ability to manage sessions without giving them full ClubManager access. Our architecture supports evolving toward permission-based authorization in the future by adding a permissions claim to the JWT and updating the authorization handler, but for now the role-based approach is sufficient for our use cases and keeps the system simpler to reason about.

The authorization system also integrates with the audit trail. Every authorization failure is logged with the user's ID, the endpoint they tried to access, and the role they have versus the roles required. This is valuable for security monitoring — if you see repeated authorization failures from a specific user, it might indicate they are probing for vulnerabilities or that their role assignment is incorrect.
