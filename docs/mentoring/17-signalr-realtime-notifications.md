# SignalR and Real-Time Notifications

Real-time communication is what makes a web application feel alive rather than static. When a goal is scored in a match, spectators watching on their phones should see the score update instantly — not after they refresh the page. When a session is cancelled, booked members should get a notification immediately — not the next time they happen to check the app. When a club manager approves a new member, the member should see their status change without reloading. SignalR is the technology that makes all of this possible, and understanding how we use it will help you build features that feel responsive and modern.

SignalR is a library built into ASP.NET Core that provides real-time web functionality. It abstracts over multiple transport mechanisms — WebSockets (the preferred and fastest option), Server-Sent Events (fallback for environments that do not support WebSockets), and Long Polling (the final fallback that works everywhere). The client and server negotiate the best available transport automatically. In practice, modern browsers all support WebSockets, so that is what gets used in almost all cases. The fallback mechanisms exist for edge cases like corporate proxies that strip WebSocket upgrade headers.

We have two SignalR hubs in the system, each serving a different purpose. The NotificationHub handles general notifications — things like "your booking was confirmed," "your payment was received," "a new event was published." These are personal notifications directed at specific users. The MatchCentreHub handles live match scoring — broadcasting score updates, match events (goals, wickets, cards), and commentary to everyone watching a specific match. These are public broadcasts that anyone can subscribe to.

The NotificationHub works on a per-user basis. When a user connects to the hub (which happens automatically when they log in to the Angular frontend), they are added to a group identified by their user ID. When the system needs to send them a notification, it sends a message to that user's group. Only they receive it. This is how personal notifications work — the backend can push a message to a specific user's browser without them needing to poll or refresh.

The connection lifecycle is important to understand. When the Angular app starts, the SignalR client (from the @microsoft/signalr package) establishes a persistent connection to the hub endpoint at /hubs/notifications. This connection stays open for the duration of the user's session. If the connection drops (network interruption, server restart), the client automatically attempts to reconnect with exponential backoff. During the disconnection period, any notifications that would have been sent are lost — SignalR does not queue messages for offline clients. For critical notifications, we also store them in a database table so the user can see them when they reconnect.

Authentication for SignalR connections uses the same JWT token as regular API calls, but with a twist. WebSocket connections cannot send custom headers in the initial handshake (this is a browser limitation), so the token is passed as a query string parameter instead. The Angular client appends the access token as an "access_token" query parameter when establishing the connection. On the server side, the JWT Bearer authentication is configured with a special OnMessageReceived event handler that checks for this query parameter on requests to /hubs/* paths and extracts the token from there instead of the Authorization header.

The MatchCentreHub is designed for public broadcasting. When a match is in progress, anyone can connect and subscribe to updates for that match — they do not need to be authenticated (though authenticated users get a richer experience). The hub uses SignalR groups to manage subscriptions. When a client wants to watch a specific match, they call a hub method like JoinMatchGroup with the match ID. This adds their connection to a group named after the match. When a score update happens, the server broadcasts to that group, and all connected clients watching that match receive the update simultaneously.

The flow for live scoring works like this. A club manager or designated scorer has the scoring interface open on their device — a phone, tablet, or laptop at the ground. When something happens in the match (a goal is scored, a wicket falls, a card is shown), they tap the appropriate button in the UI. This sends a regular HTTP POST request to the Competitions API, which updates the Match entity in the database. After the database update succeeds, the handler publishes the update through the MatchCentreHub to all connected spectators. The update includes the new scores, the type of event (goal, wicket, card), the timestamp, and optionally a text description ("Smith scores from close range, 2-1").

On the frontend, the Match Centre page establishes a SignalR connection when opened and subscribes to the relevant match group. It renders a live scoreboard that updates in real time — scores change, event timelines grow, and commentary appears without any page refresh. The experience is similar to following a match on a sports news website, but specific to the club's own matches. For cricket, the scoreboard shows runs, wickets, overs, and the current batting partnership. For football, it shows goals and the match clock. The sport-specific rendering is handled by the frontend based on the club's sport configuration.

The INotificationService in the shared infrastructure provides a clean abstraction for sending notifications from any module. When the Sessions module cancels a session, it calls notificationService.SendToUserAsync(userId, notification) for each booked member. The notification service handles the SignalR broadcast internally. The calling module does not need to know about SignalR, hubs, groups, or connections — it just says "send this notification to this user" and the infrastructure handles the rest.

Notification types are structured to allow the frontend to render them appropriately. Each notification has a type (BookingConfirmed, PaymentReceived, SessionCancelled, EventPublished, MembershipExpiring, etc.), a title, a message body, a timestamp, and optionally a link to the relevant page in the app. The frontend notification component uses the type to choose an icon and colour, and the link to make the notification clickable — tapping "Your booking for Tuesday Nets was confirmed" takes you to the session detail page.

The notification bell in the app header shows a count of unread notifications. Clicking it opens a dropdown showing recent notifications. Each notification can be marked as read (which decrements the counter) or clicked to navigate to the relevant page. This is a standard notification UX pattern that users expect from modern applications.

For the offline scenario — when a user is not connected to SignalR because they closed the app or lost network — notifications are persisted to a database table. When the user reconnects, the frontend fetches any notifications that were created since their last connection. This ensures they do not miss important updates just because they were offline for a few hours. The database serves as a durable store while SignalR provides the real-time push.

The scalability consideration for SignalR is worth mentioning even though it is not an immediate concern for development. In a single-server deployment, SignalR connections are managed in memory — all connected clients are on the same server, so broadcasting is straightforward. If you scale to multiple servers behind a load balancer, you need a backplane — a shared message bus that allows one server to broadcast to clients connected to other servers. Redis is the standard backplane for SignalR. Our infrastructure already includes Redis (with a graceful fallback to in-memory if Redis is unavailable), so adding the SignalR Redis backplane for production scaling is a configuration change rather than an architectural change.

The WebhookService in the shared infrastructure provides an alternative notification mechanism for external integrations. While SignalR pushes to browser clients, webhooks push to external systems. A club might want to receive a webhook when a new member registers (to trigger a workflow in their CRM) or when a payment is received (to update their accounting system). Webhooks are HTTP POST requests to a configured URL with a JSON payload describing the event. They complement SignalR rather than replacing it — SignalR is for real-time UI updates, webhooks are for system-to-system integration.

When building features that need real-time updates, the pattern is consistent. In your command handler, after the database operation succeeds, inject INotificationService and call the appropriate send method. For personal notifications, use SendToUserAsync. For broadcast notifications (like match updates), use the hub directly through IHubContext. The frontend subscribes to the relevant SignalR events in the component that needs to react, and updates its state when messages arrive. The Angular SignalR service in the core services folder handles connection management, reconnection, and message routing.
