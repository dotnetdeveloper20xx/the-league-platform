# Subscriptions and Feature Gating

The Subscriptions module is what makes this a SaaS platform rather than just a club management tool. It implements the business model that generates revenue for the platform owner — tiered pricing where clubs pay a monthly fee based on the features and capacity they need. This module handles subscription tiers, feature limits, trial periods, upgrades, downgrades, and dunning (the automated process of retrying failed subscription payments). Understanding this module is essential because it affects what every other module can do — feature gating means that a club's subscription tier determines which features are available to them.

There are four subscription tiers, each designed for a different size and sophistication of club. The Free tier is for clubs just getting started or evaluating the platform. It provides basic member management, simple session scheduling, and limited storage — enough to run a very small club but with clear limitations that encourage upgrading. The Starter tier at twenty-nine pounds per month adds event management, basic payment processing, and increases the member limit to two hundred. It is designed for small clubs that need more than spreadsheets but do not need advanced features.

The Pro tier at seventy-nine pounds per month is the sweet spot for most established clubs. It unlocks everything: competitions, analytics, communications campaigns, equipment management, programs, and unlimited members. Most clubs that are serious about using the platform end up on Pro. The Enterprise tier at one hundred and ninety-nine pounds per month adds white-labelling (the club's own branding with no mention of The League), custom domain support, priority support, API access for integrations, and dedicated account management. This is for large clubs or multi-club organizations.

The SubscriptionTierConfig entity defines what each tier includes. It is essentially a feature matrix stored in the database rather than hardcoded. Each config record specifies the tier name, the member limit (fifty for Free, two hundred for Starter, unlimited for Pro and Enterprise), the storage limit in gigabytes, the monthly SMS credit allocation, and boolean flags for each gated feature (competitions enabled, analytics enabled, campaigns enabled, and so on). Storing this in the database rather than in code means tier configurations can be adjusted without deploying new code — you could add a new feature flag or change a limit through a database update.

The FeatureGateService is the runtime enforcement mechanism. Other modules call this service before performing gated operations. The service looks up the club's current subscription tier, retrieves the tier configuration, and checks whether the requested operation is allowed. If a club on the Starter tier tries to create a competition, the FeatureGateService returns a failure result indicating that competitions require the Pro tier. The calling handler then returns an appropriate error to the user, and the frontend can display an upgrade prompt.

Feature gating is implemented as explicit checks in command handlers rather than as middleware or a pipeline behaviour. This is a deliberate design choice. Not every operation in a module is gated — viewing data is typically always allowed, while creating or modifying data might be restricted. A club on the Free tier can still see the competitions section in the UI (so they know the feature exists), but when they try to create a competition, the handler checks the gate and returns a "please upgrade" response. This approach gives us fine-grained control over exactly what is gated and provides a better user experience than hiding features entirely.

The member limit is the most commonly hit gate. When a club manager tries to add a new member and they are on the Starter tier with a two-hundred-member limit, the handler calls FeatureGateService.CheckMemberLimit(). The service counts the current active members for the club and compares against the tier's limit. If they are at or over the limit, the operation is denied with a clear message: "Your Starter plan allows up to 200 members. You currently have 200. Please upgrade to Pro for unlimited members." This is a soft sell built into the product experience.

Storage limits work similarly. Each club has a storage allocation for uploaded documents, profile photos, and attachments. The Documents module checks the storage gate before accepting file uploads. If the club has used their allocated storage, uploads are rejected until they either delete existing files or upgrade their tier.

Trial periods are a critical part of the acquisition funnel. When a new club signs up for the platform, they automatically get fourteen days of Pro-level access for free. During the trial, all features are unlocked and all limits are raised to Pro levels. This lets the club experience the full platform before committing to a paid plan. The trial logic is simple — the subscription record has a TrialStartDate, and the FeatureGateService checks whether the current date is within fourteen days of that start date. If it is, Pro-level access is granted regardless of the actual tier.

When the trial expires, the club is downgraded to the Free tier unless they have entered payment details and selected a paid plan during the trial. The downgrade is not immediate — there is a grace period of a few days with reminder emails. If they still have not upgraded after the grace period, the downgrade takes effect and gated features become unavailable. Their data is not deleted — it is still there, just inaccessible until they upgrade. This is important for retention — a club that downgrades and later decides to upgrade should find all their data intact.

Upgrades and downgrades change the club's tier immediately (for upgrades) or at the end of the current billing period (for downgrades). When a club upgrades from Starter to Pro, they get immediate access to Pro features and are charged the prorated difference for the remainder of the current billing period. When they downgrade from Pro to Starter, they retain Pro access until the end of the period they have already paid for, then the downgrade takes effect. This is fair — they paid for Pro this month, so they should get Pro this month.

Dunning is the process of handling failed subscription payments, and it is more nuanced than you might think. When a club's monthly subscription payment fails (card expired, insufficient funds, bank declined), the system does not immediately downgrade them. That would be terrible user experience — a card expiring should not instantly lock a club out of their management platform. Instead, the system enters a dunning flow.

The dunning flow retries the payment at configured intervals. The first retry happens one day after the initial failure. If that fails, the second retry happens three days later. If that fails, the third retry happens seven days later. Between retries, the club receives email notifications explaining that their payment failed and asking them to update their payment method. The notifications escalate in urgency — the first is gentle ("heads up, your payment failed"), the second is firmer ("please update your card"), and the third is a final warning ("your subscription will be cancelled if payment is not received").

If all retry attempts fail, the subscription is marked as past-due and eventually cancelled. Cancellation triggers a downgrade to the Free tier, with the same data-preservation approach described above. The club can reactivate at any time by updating their payment method and selecting a plan — their data will still be there.

Each retry attempt is logged with the timestamp, the result (success or failure), and the error message from the payment provider. This audit trail is important for customer support — if a club contacts support saying "why was my subscription cancelled?" the support team can see exactly what happened: three failed payment attempts over eleven days, three notification emails sent, no response from the club.

The seed data creates four subscriptions — one per club. Teddington and Richmond are on Pro at ninety-nine pounds per month (the seed uses slightly different pricing from the tier definitions for historical reasons). Highbury is on Starter at forty-nine pounds per month. The MCC is on Enterprise at two hundred and forty-nine pounds per month. All are active with current billing periods. This lets you test feature gating by logging in as different club managers and observing which features are available based on their club's subscription tier.

The billing cycle for platform subscriptions is always monthly. This is separate from membership billing cycles, which can be annual, quarterly, monthly, or any other period. Do not confuse the two — platform subscriptions are what clubs pay to The League for using the software. Membership fees are what members pay to their club for being a member. They are completely independent financial flows handled by different modules.
